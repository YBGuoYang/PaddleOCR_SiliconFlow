# PaddleOCR_SiliconFlow 更新计划

日期：2026-03-26

## 目标

采用方案 B，只保留“托盘 + 快捷键”这一条主产品形态，删除或下线简版窗口入口，把当前项目从“脚本集合”整理成“可维护的小工具”。

## 当前结论

- 当前代码可以编译，但工程边界较松。
- 项目存在双入口分叉，配置和用户体验不一致。
- `requirements.txt` 当前不可靠，后续需要重写。
- 后续开发和测试统一基于项目内 `.venv`。

## 执行原则

- 单一入口：只保留热键托盘版为正式入口。
- 单一配置源：API Key、热键、通知等只走一套配置模型。
- 单一核心流程：截图、OCR、结果解析、通知、剪贴板都抽到公共模块。
- 先稳定再美化：先收口结构和错误处理，再处理界面与发布体验。
- 每阶段都要可运行，避免一次性大重构。

## 分阶段计划

### 第 0 阶段：环境与基线

- 固定当前开发测试环境为项目内 `.venv`。
- 清点当前实际运行依赖与无效依赖。
- 重写 `requirements.txt`，只保留真实需要的包。
- 补充最小开发说明，明确推荐 Python 版本。

验收标准：
- 新环境可在 5 分钟内完成安装。
- `python -m py_compile` 可通过。
- `pytest --version` 可用。

### 第 1 阶段：产品收口

- 废弃 `scripts/screenshot_ocr_simple.py` 对应的正式入口。
- 明确主入口为托盘热键版。
- 统一 `run.bat`、`run_hotkey.bat`、打包脚本的行为与文案。
- 移除与主路径不一致的说明和分支逻辑。

验收标准：
- 用户只看到一套启动方式。
- API Key 配置、热键配置、截图触发方式全部一致。

### 第 2 阶段：核心模块拆分

- 新建核心包，例如 `src/screenshot_ocr/`。
- 拆分配置层、OCR 客户端层、截图层、通知层、剪贴板层、应用编排层。
- UI 层只负责交互，不再直接处理 OCR 请求和文件清理。
- 提取公共的结果解析和错误处理逻辑，消除简版/托盘版重复代码。

建议目标结构：

```text
src/screenshot_ocr/
  __init__.py
  main.py
  app.py
  config.py
  capture.py
  ocr_client.py
  clipboard.py
  notifier.py
  tray_ui.py
```

验收标准：
- 热键版核心流程只经过一套服务实现。
- 重复逻辑明显下降。
- 模块职责可以直接从文件名判断。

### 第 3 阶段：行为修正与稳定性

- 修复通知开关保存但不生效的问题。
- 处理 `ImageGrab` 引入、临时文件清理、异常吞掉等问题。
- 删除桌面调试截图这类开发残留逻辑。
- 清理“文案支持但代码不支持”的功能，例如鼠标侧键。
- 为 OCR 请求增加更明确的超时、重试和错误提示。

验收标准：
- 开关项行为与界面文案一致。
- 临时文件不会泄漏。
- 错误能定位到配置、网络、截图或 OCR 请求具体阶段。

### 第 4 阶段：测试补齐

- 为配置加载、OCR 响应解析、文本去重逻辑补单元测试。
- 为关键业务路径补最小集成测试。
- 对外部依赖做 mock，避免测试直接调用真实 API。
- 为托盘和截图相关不可自动化部分补手工测试清单。

优先测试点：

- 配置文件缺失、损坏、字段缺省时的加载行为。
- OCR 接口返回空文本、异常结构、超时、非 200 状态码。
- 文本去重策略是否误删内容。
- 热键配置切换后是否正确重载监听。

验收标准：
- 关键纯逻辑模块具备可重复运行的测试。
- 后续改动可以靠测试快速发现回归。

### 第 5 阶段：打包与发布整理

- 精简 `build.bat`，减少手写 hidden imports。
- 明确打包后的配置目录、首次启动行为和分发方式。
- 校验 README、启动脚本、打包脚本三者描述一致。
- 整理版本发布说明，形成最小发布流程。

验收标准：
- 可稳定产出可运行的 EXE。
- 用户首次使用路径简单且文档一致。

## 建议执行顺序

1. 重写依赖和启动脚本基线。
2. 删除简版入口，统一正式入口。
3. 拆核心模块，建立单一配置模型。
4. 修正功能行为和异常处理。
5. 补测试。
6. 整理打包与 README。

## 非目标

- 暂不增加新的 OCR 服务商适配。
- 暂不做复杂 UI 美化。
- 暂不支持多平台，继续以 Windows 为主。

## 第一轮改造建议产出

- 一个可运行的单入口版本。
- 一个干净的 `requirements.txt`。
- 一套统一配置读写逻辑。
- 一批覆盖核心解析逻辑的基础测试。
